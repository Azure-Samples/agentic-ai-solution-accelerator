# Supported Customization Boundary

> **Audience:** partner engineers, partner architects, MSFT field, support intake.
> **Purpose:** define exactly what partners MAY and MAY NOT change without losing attestation + support.

This is a **normative** document. The [partner playbook](partner-playbook.md) is the **procedural** complement — it tells you *how* to work within these boundaries.

---

## 1. Supported fast path

The accelerator ships **azd templates for the 5 blessed bundles** as the supported fast path:

| Bundle | `azd-templates/<bundle>/` |
|---|---|
| `sandbox-only` | azd template + guided-demo overlay |
| `retrieval-prod` | azd template |
| `retrieval-prod-pl` | azd template + private-link overlay |
| `actioning-prod` | azd template |
| `actioning-prod-pl` | azd template + private-link overlay |

`azd up` against these templates is the only path that qualifies for the "hours for sandbox, days for pilot, weeks for prod" commitment (§4 of `docs/partner-playbook.md`).

## 2. What you MAY NOT change (invariants)

Changing any of the following **breaks attestation** and places the engagement outside the supported envelope:

### 2.1 Platform invariants
- **Baseline package versions** in `baseline.lock.yaml` — may only move via `baseline upgrade`.
- **Bicep module semantics** under `delivery-assets/bicep/modules/` — wiring, identity, KV references, diagnostic settings, network posture. These are authored once in the accelerator; partners must not fork or patch-in-place.
- **Azure Monitor alert rules + Action Groups** generated by `baseline materialize alerts` — values adjustable via `*.override.yaml`; structure not.
- **App Insights telemetry schema** emitted by T1 core — agent/tool/cost/eval event shapes are fixed.
- **Kill-switch / cost-ceiling enforcement** in T1 core — may not be bypassed or stubbed.
- **HITL + side-effect-tool wiring** from `baseline-hitl` + `baseline-actions` for `actioning-*` bundles.

### 2.2 Attestation invariants (RAI tuple)
- **Model ID + alias** — drift triggers re-attestation + RAI IA review.
- **Tool set** (names, kinds, side_effect flags, configs) — drift triggers re-attestation.
- **Side-effect tool names** explicitly — any additions require `actioning-*` bundle + HITL.
- **Grounding source manifest** (source IDs, types, URL patterns, ACL models, classifications) — drift triggers re-attestation.
- **Agent topology** (agents list, edges, HITL points, a2a cap) — drift triggers re-attestation.
- **Instructions / prompt templates** — drift triggers re-attestation (tracked via `instruction_hash`).

### 2.3 Bundle + profile invariants
- Bundle must be one of the 5 blessed values. **New bundles require governance board approval.**
- Profile must match bundle per the matrix in `spec.schema.json`.
- Side-effect tools require `actioning-*` bundle; validator enforces.

### 2.4 Governance invariants
- Waivers: max 5 active per repo, 90d SLA, high-severity requires governance board.
- RAI IA: max 180d validity, CODEOWNERS + immutable UUID required.
- Qualification (Path B/C): GitHub review + Rekor UUID required.

## 3. What you MAY change (customer business layer)

Everything below is partner + customer territory, vibecoded with GitHub Copilot in VS Code. The accelerator ships scaffolding + examples, not finished code.

### 3.1 Business logic (free-form)
- **Agent prompts / instructions** — authored per customer, snapshotted under `rai/snapshots/` at attestation time.
- **Tool implementations** — custom MCP servers, function calls, REST wrappers.
- **Grounding pipelines** — indexing, chunking, ACL-aware retrieval glue.
- **Domain-specific transforms** — business validation, response shaping.
- **Evaluation datasets** — customer-labeled quality + red-team suites.
- **Scenario-specific workflows** — within topology cap (2-agent a2a).

### 3.2 Parameterizable (via `.override.yaml` companion)
- **SKU sizes, region selection** — within allowed profile SKUs (see `patterns/waf-alignment/`).
- **Alert thresholds** — over the baseline minimums.
- **Dashboard visualizations** — append-only.
- **Cost ceiling** — may be lower than platform default; not higher than customer-approved max.
- **Identity model details** — within the 3 `identity_model` enum values.

### 3.3 Appendable (layered on top)
- **Additional KPIs** in Spec.
- **Additional grounding sources** (re-attestation triggered).
- **Additional quality eval metrics** (regression threshold still enforced).

## 4. Non-attested customization (BYO-IaC escape hatch)

Some customers mandate BYO-IaC (Terraform, in-house landing zone, non-azd pipelines). The accelerator does not reject these engagements, but they are **non-attested unless re-qualified**:

### 4.1 What must remain invariant
- **Topology hash** — agents + edges + HITL + a2a cap per Spec.
- **RAI tuple** — model, tools, instructions, grounding manifest, data classification.
- **Control set semantics** — kill switch, cost ceiling, content sanitization, retry policy, SSE streaming, auth via MI, secrets via KV, telemetry via App Insights.
- **Bicep module contracts** — see `delivery-assets/bicep/modules/*/MODULE_CONTRACT.md` (published per module): interface, identity model, network posture, diagnostics, outputs.

### 4.2 Requirements for re-qualification
- Partner must submit a **BYO-IaC conformance package** to the governance board:
  - Mapping of partner's IaC to every Bicep module contract
  - Proof of equivalent control set
  - Proof of equivalent telemetry + cost accounting
  - Sign-off from MSFT field CTO
- Governance board reviews on quarterly cadence (or 5-BD SLA for active lighthouse engagements).
- Re-qualification is valid for 180 days and tied to a specific partner org + IaC version.

### 4.3 What re-qualification does NOT grant
- Emergency security patch lane coverage (partner owns their own patch propagation).
- Automatic `baseline upgrade` compatibility (partner owns their own upgrade transaction model).
- Default azd template improvements (partner owns backporting).

**Recommendation:** BYO-IaC is a real customer ask but adds material delivery + support overhead. For v1, favor moving customers to an azd-deployed subscription for the agentic solution specifically, even if their broader estate is BYO-IaC.

## 5. Bundle variant requests

Partners WILL request bundle variants (e.g., "retrieval-prod but with cache," "actioning-prod without HITL for a narrow tool set," "workflow-prod for a 3-agent graph").

**Rule (v3.4.2):** variants are expressed via **Spec parameters, profile variants, and optional T2/T3 extensions**, NOT new bundle rows.

| Ask | Resolution |
|---|---|
| "retrieval-prod with cache" | T3 `baseline-cache` added; perf SLOs explicitly exclude cache-dependent optimizations |
| "actioning-prod with narrow HITL" | HITL scope configured per-tool in Spec; baseline-hitl still required for bundle |
| "3-agent workflow" | Governance board approval required; re-evaluates a2a cap |
| "retrieval-prod in non-PL region" | Covered by `prod-standard` profile; no new bundle |
| "cost-only-cap bundle" | Spec cost_ceiling adjusts; no new bundle |

Governance board approves new bundles **only** if all of: (a) material WAF posture change, (b) material RAI posture change, (c) engagement volume justifies dedicated support scope.

## 6. Enforcement

- **Validator (`tools/validate-repo.py`)** — fails closed on invariant violations.
- **Reconcile (`baseline reconcile`)** — detects drift vs attested state (block / warn / auto-adopt classes; see `compatibility/drift-classification.yaml`).
- **Support intake** — rejects tickets where deployed state doesn't match attested lockfile.
- **Attestation issue** — refuses to sign a snapshot that violates invariants.

---

## 7. Change process for invariants themselves

Invariants in this document are not static; the accelerator evolves. Changing an invariant (adding, removing, widening, narrowing) requires:

1. PR with proposed invariant change.
2. Rubber-duck critique (internal, documented).
3. Governance board approval (high-severity waiver authority).
4. Schema version bump (if Spec/qualification schema affected).
5. Release manifest entry (`compatibility/releases/<version>.yaml`).
6. Expand-contract migration plan (if existing customers affected).

See `compatibility/upgrade-transaction-model.md`.
